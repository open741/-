# 评估综合分析报告需求说明文档

## 1. 功能概述
评估综合分析报告模块旨在通过整合学员的多份评估报告数据，利用 AI 技术自动生成一份全面的综合分析文档，为后续的教学计划（IEP）提供科学依据。

## 2. 核心流程与功能描述

### 2.1 评估报告记录
*   **学员选择**：
    *   用户需首先选择学员。
    *   **联动机制**：重新选择学员或更换学员后，下方已生成的所有数据（基本情况、评估结果、分析结论等）必须自动清空。
*   **报告选取**：
    *   系统展示该学员名下所有类型的历史评估报告。
    *   **选取规则**：每一类评估报告只能选择一份（通常为最新的一份）。
    *   **操作反馈**：支持重新选择或取消已选报告。
    *   **联动机制**：重新选择或取消任何一份报告后，下方所有已生成的评估数据及 AI 分析结果必须自动清空。

### 2.2 评估时的基本情况
*   **功能定义**：描述学员在进行评估过程中的表现、配合程度及外部环境等基础背景。
*   **生成逻辑**：由 AI 根据所选评估报告中的原始记录数据自动聚合生成。
*   **AI 提示词 (Prompt)**：
```markdown
# 角色: 专业的特殊教育评估师和康复干预专家

# 任务: 
你现在的任务是根据系统提供的一份【已按领域】分类好的儿童评估报告数据，撰写一段客观、专业、连贯的“评估时的基本情况”综合分析文字。

# 背景与上下文:
系统可能收集了一种或多种不同的评估报告（如孤独症评估、0-6岁发育评估、CPEP-3、感统评估等），但输入给你的数据已经过程序预处理，被精准归类到了IEP的8大领域中。你需要综合这些指标（如百分比分数、发育商、智龄、发展年龄、得分等），结合儿童的【实际月龄】，进行专业、客观、有温度的综合分析。

# 数据处理规则:
1. **深度融合**：在同一个领域内，如果包含来自不同报告的多个数据（如既有“0-6岁语言发育商”又有“CPEP-3语言理解百分比”），请不要像机器一样简单罗列分数，而是要交叉比对这些数据，将其转化为对儿童当前该领域真实能力的具象化描述。
2. **年龄基准对比**：密切关注数据中的“智龄”或“发展年龄”，并将其与儿童的“实际月龄”进行对比。客观且清晰地指出发育水平（如：显著落后、临界、与生理年龄相符）。
3. **指标科学解读**：正确解读各类分数的临床意义（例如：发育商≥130为优秀，80-109为中等，<70为发育障碍）。
4. **行文基调**：保持客观、专业。切忌使用绝对化或带有歧视色彩的词汇。
```

### 2.3 评估结果（分领域）
*   **展示逻辑**：
    *   根据选取的报告类型，动态展示其包含的康复领域。
    *   **过滤规则**：只显示所选报告中实际包含且已进行过评测的领域。
*   **生成逻辑**：
    *   **通用领域**：每个领域提供独立的 AI 生成按钮，点击后根据该领域的评估数据单独生成分析结果。
    *   **孤独症报告例外处理**：
        *   内容直接引用报告中的“评估总结”。
        *   点击 AI 分析按钮时，直接调用对应评估报告模块内的 AI 生成功能。
*   **AI 提示词 (Prompt)**：
```markdown

# 任务:

# 背景与上下文:

# 数据处理规则:
1. **精准聚焦**：只针对本次传入的【评估领域名称】进行解读，绝不可延伸、臆测或罗列其他领域的内容（例如：传入的是“大运动”，就只描述大动作能力相关的表现）。
2. **年龄基准对比**：如果传入的领域数据中包含“智龄”或“发展年龄”，必须将其与儿童的“实际月龄”进行直观对比。
3. **指标科学转化**：将冰冷的分数转化为专业的临床描述。正确解读各类分数的意义（例如：发育商≥130为优秀，80-109为中等，<70为发育障碍；标准分或百分比代表的适应程度）。
4. **行文基调**：
    - 拒绝干瘪的罗列分数。
    - 将数据融于描述性语言中（如：“在【特定领域】领域，儿童当前智龄评估约为X个月，整体发育商为Y，处于……”）。
    - 语言保持客观中立，客观陈述该项能力的具体水平及核心受限点，切忌使用绝对化或带有歧视色彩的词汇。

- 来源报告名称：{{报告名称}}（例如：0岁~6岁儿童发育行为评估报告）
- 评估领域名称：{{评估领域名称}}（例如：大运动）
- 领域评估数据：
{{此处传入该单一领域相关的具体指标数据，例如：{"智龄": 24, "发育商": 68, "评价": "发育迟缓"} }}

# 输出格式要求:
请严格遵守以下输出规则：
1. 仅输出一段纯文本的分析段落，字数控制在 80-150字 左右。
2. 绝对不要输出任何标题（如“评估结果”、“大运动：”等）。
3. 绝对不要使用任何加粗、列表等 Markdown 格式。
4. 直接输出正文内容，以便代码层面能够直接无缝读取并拼接至报告模板。
```

## 3. 交互与 UI 要求
*   **表格化展示**：评估结果需采用表格形式，包含“所属报告”、“领域”、“评估结果”三列，支持单元格合并。
*   **吸底操作栏**：取消和提交按钮需固定在页面底部，不随页面滚动，且不得遮挡左侧菜单栏。
*   **响应式布局**：页面需支持大屏幕显示，确保在 1280px (max-w-7xl) 下拥有良好的阅读体验。
